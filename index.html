Como configurar terminal

General settings

  • git --version (check the version of git)
  • git config --global user.name "Erick Chavez" (Global configuration: user)
  • git config --global user.email erickchavezescobar@gmail.com (Global configuration: email)
  • git config --global init.defaultBranch main (Global configuration: main instead of master)
  • git config --global core.editor "code --wait" (Global configuration: VS)
  • git config --global -e (Global configuration: summary in VS, close it to reactivate terminal)
  • git config --global core.autocrlf input(Al agregar saltos de línea se agrega el caracter especial Line Feed "LF". Al subir el código no se agrega el caracter Carriage Return "CR", pero aún así git debe eliminar el caracter de forma automatica a CRL )
  • git config -h (Retorna listado de todas las opciones que podemos agregar a la configuración)

UPDATING GIT

  • brew install git (installation)
  • export PATH=/usr/local/bin:$PATH (Run it)
  • rm '/usr/local/bin/git' (Run to delete old git)
  • brew link --overwrite git (Run to overwrite it)

GIT FLOW

  • git add [file to be added] [.] adds all, [*.txt] adds all txt files
  • git status (show which files are tracked/commited)
  • git commit -m "[message]" (Adds message to the files which are ready to be pushed)
  • git rm [file to be removed] ("ls" to list all files and the choose one to be removed)
  • git restore --staged [file to be restored] (Unstages the file which was removed)
  • git restore [file to be brought back] (Now the file is available again OR is back to its previous status: UNDO)
  • git mv [file to be renamed] [file with new name] (To rename a file and stage changes at the same time!)

WORKING WITH FILES

  • Create a file .env with sensitive information
    create a file .gitignore with the routes for the files to be ignored. (Ignoring files to never be included in the git REPO. Config files for local computer: variables de entorno [passwords])
  • Add the names of the files and folders to be hidded in the .gitignore file (For example: .env , node_modules)
  • git add .gitignore & git commit -m "..." (When checking with status you'll see that the only file available to be added and commited is .gitignore, removing the files listed inside it.)
  • git status -s (Muestra una versión corta del status (RED = unstaged / GREEN = staged or added & ready to be commited). M = modified, ?? = untracked, A = .add or added)
  • git diff (When changes are made to a file but are not added (.add) we can check how they changed using git diff)
  • git diff --staged (Only works when changed have been staged with "git add". Now will see which changes have been staged)
  • git log (Shows all changes, author, date, hash.)
  • git log --oneline (Shows a shorten version of the hash and the message commited)
  • git log -p (Muestra la versión anterior y los cambios)
  • cat [file name] (Shows the content of the file in the TERMINAL)

Branches

  • git branch -m master main (Renaming MASTER to MAIN in local repositories)
  • git push -u origin main (Renaming the Remote master Branch: Make sure your current local HEAD branch is still "main" when executing the command)
  • git push origin --delete master (Remove the old "master" branch on the remote.)
  • git pull origin [branch-name] (Descarga las versión actualizada de otra rama a la rama donde nos encontremos.)
  • git branch (Lists branches and shows you where you are.)
  • git branch [branch-name] (Creates a new branch. Make sure the name is separated by a dash)
  • git checkout -b [name of the new brach] (Creates and switches to new branch. You can name them using feature/branchName)
  • git branch -m [new-name] (Cambiar el nombre de una rama. Ubicarse primero en la rama)
  • git branch -m [old-name] [new-name] (Cambiar el nombre de cualquier rama desde la rama "main".)
  • git branch -d [branch-to-be-deleted] (Eliminar ramas locales desde main)
  • git merge [branch-to-be-merged] (First git checkout [destination-branch], then git merge)
  • git merge --abort (Aborts a merge, when a conflic occurs)
  • git checkout -b [newBranch] & git push -u origin [newBranch] (To creat a new branch without merging it, but having it available in GITHUB)

Commits

  • git commit --amend (Modify the message of the last commit. A tab pops in VS and you can edit it there.)
  • git reset --soft HEAD~1 (Deshacer el último commit. Con SOFT no se modifican los archivos. HEAD es el último commit que se realizó. Con HARD se eliminan los cambios.)

RESOLUCIÓN DE CONFLICTOS

  • git merge --continue (Permite continuar el proceso de merge. También funciona solo usar con git commit y git push.)

GITHUB REPOSITORIES

  • git clone [github-url] (Clones a repository)
  • git remote (Muestra el nombre del repositorio remoto: origin.)
  • git remote -v (Muestra detalles de origin: fetch [obtener los cambios realizados en el repo] & push [enviar al repo remoto los cambios realizados locales])
  • git push origin main (Realiza un commit antes de enviar los cambios al repo remoto en GitHub. If GitHub isn't connected you'll have to sign in or use a TOKEN [developer settins, personal access tokens, tokens (classic), select scope: repo & gist!])
  • git pull origin main (PULL descarga el contenido de un repo remoto e inmediatamente actualiza el repo local. Hace un MERGE.)
  • git fetch origin (FETCH verifica los cambio realizados en el repo remoto sin combinarlo al repo local, desde la última vez que se hizo git pull.)
  • git checkout origin/main (Muestra los cambios que se hicieron en la rama remota antes de decidir si queremos mezclarlos. Para regresar a la rama de trabajo local: git checkout main)
  • CLICK on "fork", (select) copy "main" branch only, click "create fork", *optional* sync fork (to keep repo updated), clone file in terminal (main). (FORK = Bifurcar: crear una copia de un repo remoto en GitHub). You can contribute to the original file with PULL REQUESTS
    • After making changes, adding, commiting & pushing go to the repo in GitHub and click "contribute", "open pull request", add comment, click "create pull request" (Pull request: solicitud de combinar tus cambios con el repo original.)
    • Click on: settings, pull requests, [request-link], close pull request (Borrar copia difurcada.)
    • Click on: repo settings, eliminar repo (Después de borar la copia difurcada, eliminar el repositorio.)
  • git push origin [new-branch] (Pull requests a partir de una rama: Go to repo remoto: fork; terminal en main: clone, ls, cd folder, open in VS, new branch, edit, add, commit, **push**; )
    • Crear pull request de nuestra rama creada, al repo remoto sin compromete nuestra rama difurcada "main" modificar su main.(En GitHub: branch -> new-rama, contribute, open pull request, add comment, create pull request. )
  • GitHub: repo forked, contribute, update branch (Actualizar repositorio bifurcado --> Terminal: git checkout main, git pull origin main)

ISSUES

  • Once an issue has been opened (You can assign someone to solve it, up to ten people [assigneed] & you can add labels)
  • good first issue (Look for this label, since it's aimed to have first-time contributors solve issues.)

CREATE REMOTE BRANCHES IN GIT

  • GitHub: branches --> main, create branch: type name, click on "create branch: ..." (After creating the remote branch, refresh the site to see it.)
  • TERMINAL: cd directory, git clone (if needed), git pull origin [remote-branch-created-in-GitHub], git fetch origin, git checkout [remote-branch-created-in-GitHub] (Pull remote branch to local repo in my Terminal)
  • TERMINAL: git checkout main, git branch -d [remote-branch] --> deletes local branch, git branch -a --> shows remote branches, GitHub: you can delete the branch there --> branches, view all branches, trash icon, TERMINAL: git push origin -d [remote-branch] --> deletes remote branch. (Delete remote branch: if you haven't merged it you'll have to delete it from GitHub and local.)
